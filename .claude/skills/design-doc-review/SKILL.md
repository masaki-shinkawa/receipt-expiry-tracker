# 設計書レビュー Skill

`receipt-expiry-tracker` Wiki の機能設計書を多角的にレビューし、問題点・改善提案をフィードバックする。

## 前提情報

- Wiki URL: https://github.com/masaki-shinkawa/receipt-expiry-tracker/wiki
- レビュー対象: `Feature-{機能名}-画面設計` / `Feature-{機能名}-API設計`
- 記述言語: 日本語

## ワークフロー

TodoWrite でタスクを管理しながら以下を順に実施する。

### 1. レビュー対象を確認する

ユーザーの指示からレビュー対象の機能名を特定する。不明な場合は質問する。

### 2. 情報を収集する

以下を並行して読み込む:

- レビュー対象の設計書（画面設計・API設計）
- `REQUIREMENTS.md`（プロジェクトルート）
- `データ設計` Wikiページ
- `GAS API 共通仕様` Wikiページ
- 依存コンポーネントの設計書（設計書内の `## 依存コンポーネント` セクションを参照）
- 関連する実装コード（`src/components/` / `src/hooks/`）— 実装済みの場合のみ

### 3. レビューを実施する

以下の観点でレビューし、各項目を ✅ / ⚠️ / ❌ で評価する。

#### A. 構造チェック（必須セクションの存在確認）

| 項目 | 確認内容 |
|---|---|
| 概要 | 1〜2文で機能の目的が記述されているか |
| 受け入れ条件 | Given-When-Then 形式の表があるか（最低3件） |
| Mermaid図 | 画面設計にレイアウト図、API設計にデータフロー図があるか |
| 関連コンポーネント | 実装ファイルへの参照があるか |
| 依存コンポーネント | 他機能への依存がある場合に明記されているか |

#### B. 整合性チェック

| 項目 | 確認内容 |
|---|---|
| REQUIREMENTS.md との整合 | 要件定義の内容と設計が矛盾していないか |
| 画面設計 ↔ API設計の対応 | 画面の受け入れ条件がAPIで実現可能か |
| GAS API 共通仕様への準拠 | レスポンス形式が `{ success, data, error }` に準拠しているか |
| データ設計との整合 | フィールド名・型がスプレッドシート設計と一致しているか |
| 依存設計書の最新性 | 依存コンポーネントの設計書が存在し、最新の状態か |

#### C. 品質チェック

| 項目 | 確認内容 |
|---|---|
| 受け入れ条件の網羅性 | 正常系だけでなく異常系・境界値が含まれているか |
| Mermaid 構文 | 図が正しく記述されているか（typo・未対応構文がないか） |
| 曖昧表現の排除 | 「適切に」「必要に応じて」などの曖昧な表現がないか |
| 未定義項目 | `{プレースホルダー}` が残っていないか |

#### D. セキュリティチェック

| 項目 | 確認内容 |
|---|---|
| 入力値検証 | フロントエンドおよびGAS側でのバリデーションが考慮されているか |
| 認証・認可 | 認証不要なエンドポイントが意図的かどうか明記されているか |
| センシティブデータ | APIキー・個人情報などが設計上露出しないか |
| XSS / インジェクション | ユーザー入力をそのままDOMや外部APIに渡す設計になっていないか |

#### E. 拡張性チェック

| 項目 | 確認内容 |
|---|---|
| 将来フェーズへの影響 | Phase 2〜5 の機能追加時に大規模リファクタが必要になる設計でないか |
| ハードコーディング | 変更可能性のある値が定数化・設定化されているか |
| インターフェース設計 | コンポーネント間の結合度が高すぎないか |

#### F. 性能チェック

| 項目 | 確認内容 |
|---|---|
| 大量データ対応 | 商品数が増えた場合のUI描画・フィルタ処理が考慮されているか |
| GASリクエスト数 | 不要なAPI呼び出し（N+1問題など）がないか |
| クライアント処理 | 重い計算・画像処理がメインスレッドをブロックしないか |
| オフライン対応 | GAS未設定時のフォールバック（localStorage）が設計されているか |

### 4. 依存コンポーネントの設計書を確認する

設計書内の `## 依存コンポーネント` セクションを読み、各依存先について:

- 対応するWikiページが存在するか
- 依存先の設計書が、今回の設計と矛盾していないか
- 依存先の設計書が古い場合（今回の機能追加で影響を受ける場合）、更新が必要な箇所を指摘する
